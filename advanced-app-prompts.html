Advanced App Creation Prompts

Use these when you want more than a basic app. These prompts force structure, testing, performance, and a clean handoff.
Spec-first app build
Create a complete downloadable local application for this idea: [APP IDEA].

Before creating files, first write a short implementation spec that includes:
1. Main user goal
2. Core features
3. Non-goals so the app does not become bloated
4. Recommended technology choice and why it fits this app
5. File/folder structure
6. Performance risks
7. Error states
8. User acceptance checklist

Then create the actual runnable files. Do not give copy-and-paste code only. Use the best development approach for the app idea. Do not force batch files unless useful as optional launchers. Include a normal launcher, a debug launcher, and a README written for a non-developer.
Acceptance criteria lock
Build this app: [APP IDEA].

Treat the following as acceptance criteria. The app is not complete unless every item is handled:
- It launches without a visible command window unless debug mode is used.
- It has clear loading/progress feedback for anything that takes time.
- It stays responsive during long tasks.
- It never freezes the UI while scanning, importing, converting, loading thumbnails, or reading folders.
- It shows friendly user-facing errors and detailed debug logs separately.
- It handles empty data, huge data, missing permissions, duplicate entries, locked files, and canceled operations.
- It includes a README explaining exactly which file to launch.
- It includes a simple test checklist I can use after downloading.
Local app packaging prompt
Create a local app for [APP IDEA] and package it like something I can actually hand to another employee.

Include:
- A clear folder structure
- One obvious launch file
- Optional debug launch file
- README for normal users
- README_DEBUG for troubleshooting
- A logs folder created automatically
- A config/settings file that is safe to edit
- Default settings that work immediately
- A backup/restore plan if the app changes files or settings
- A version number shown inside the app
- A changelog file

Keep the app lightweight and choose the best technology for the job.
No-surprises technology choice
For this app idea: [APP IDEA], choose the best development approach instead of defaulting to a specific language.

Compare at least 3 realistic options briefly, then pick one. Consider:
- Windows compatibility
- Whether it needs a GUI
- Whether it needs background processing
- Whether it needs file access
- Whether it needs packaging
- Whether antivirus might flag the approach
- Whether future edits will be easy

After choosing, create the downloadable files using that approach.
State model prompt
Create this app: [APP IDEA].

Before coding, define the app state model. Include what the UI should show during:
- First launch
- Ready/idle
- Loading
- Scanning
- Paused
- Canceled
- No results
- Partial results
- Permission denied
- Error but recoverable
- Fatal error
- Completed successfully

Then implement the app so the UI always reflects the current state clearly.
Settings architecture prompt
Create this app with a safe settings system: [APP IDEA].

Requirements:
- Store user preferences in a human-readable settings file.
- Never require the user to edit code for normal customization.
- Include a reset-to-defaults option.
- Validate settings before applying them.
- If settings are broken, recover automatically and explain what happened.
- Separate visual settings from functional settings.
- Include comments or descriptions for each setting in plain language.
Release-ready app prompt
Create this as a release-ready local app, not a prototype: [APP IDEA].

Include:
- Polished UI
- Responsive long-running operations
- Progress indicators
- Logging
- Error recovery
- Keyboard shortcuts where helpful
- Accessible labels and readable contrast
- Test data or sample mode if useful
- README
- Changelog
- Version displayed in the app
- Known limitations section
- Manual QA checklist

Do not claim a feature works unless it is actually implemented in the files.

Prompts You Probably Haven’t Been Using Yet

Ask for a ruthless app review before rebuilding
Review this app idea or existing app behavior like a senior product designer and senior Windows developer.

App: [DESCRIBE APP]
Current problems: [LIST PROBLEMS]

Give me:
1. What will probably break first
2. What will feel annoying to normal users
3. What features are unnecessary
4. What hidden edge cases I am missing
5. What should be simplified
6. What should be built first
7. What should not be built yet
8. A revised prompt I can use to create the better version
Ask for a maintenance handoff
After creating the app, also give me a maintenance handoff section.

Explain in plain language:
- What each file does
- Which files are safe to edit
- Which files I should not touch
- How to change text, colors, paths, and default settings
- How to read the logs
- How to report a bug clearly
- How to roll back to a previous version
- What future improvements would be easiest

Mini Add-ons

Use this phrase often

“Do not claim a feature works unless it is actually implemented in the files.”

Use before app builds

“Define the acceptance checklist first, then build to pass it.”

Use for weird bugs

“Rank likely causes by evidence, not vibes.”

Use these as strong constraints, checklists, and acceptance criteria. No prompt can override safety rules, tool limits, or unavailable information, but these are designed to remove ambiguity and make the output much more likely to match what you intended.
