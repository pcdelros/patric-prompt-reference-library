Debugging, Performance, and Large Folder Prompts

Best for your photo/video and drive-mapping apps: these prompts force background work, lazy loading, caches, cancellation, and useful logs.
Crash reproduction prompt
Help me debug this app problem: [DESCRIBE PROBLEM].

Do not guess randomly. Give me:
1. Most likely causes ranked by probability
2. What evidence would confirm each cause
3. What logs or debug output the app should collect
4. A safe patch plan
5. The exact files/functions likely needing changes
6. A test checklist to confirm the fix

Then update the app files to include better logging and the fix.
Large folder performance prompt
This app must handle massive folders with photos/videos without freezing.

Implement performance protections:
- Scan folders in a background worker/thread/process
- Update progress continuously
- Allow cancel/pause/resume if realistic
- Load results in batches
- Use lazy loading for thumbnails
- Cache thumbnails on disk
- Do not decode full-size images just to make thumbnails
- Do not load all videos into memory
- Use virtualized scrolling for large grids
- Keep the UI responsive while scanning
- Show partial results as they are found
- Log scan speed, file count, skipped files, and errors

Design for tens of thousands of files, not just a small test folder.
Progress bar that actually means something
Add a progress system that is honest and useful.

Requirements:
- Show current phase: scanning, indexing, thumbnailing, loading, complete
- Show file count processed out of total when total is known
- If total is unknown, show an indeterminate progress indicator plus live count
- Show current folder or current file type, but avoid flashing too fast
- Include cancel button for long scans
- Keep the progress area visible but not annoying
- Write progress events to the debug log
Thumbnail performance prompt
Fix thumbnail loading so it feels fast and smooth.

Requirements:
- Generate thumbnails asynchronously
- Prioritize visible items first
- Use a disk cache so thumbnails do not regenerate every launch
- Use placeholders while loading
- Avoid blocking scrolling
- Limit concurrent thumbnail jobs so the PC does not get overwhelmed
- Store cache safely and allow clearing it from settings
- Videos should show a generated preview frame when possible, not a blank icon
- If a file fails thumbnail generation, show a friendly fallback icon and log the reason
Freeze/not responding prompt
The app goes Not Responding during [ACTION]. Fix the architecture so long work never runs on the UI thread.

Requirements:
- Identify every long-running task
- Move long-running tasks off the UI thread
- Add progress updates using safe UI messaging
- Add cancellation support where practical
- Add timeouts for risky operations
- Add try/catch error handling with friendly messages
- Add detailed logs for developers
- Keep the app usable while work continues
- Include a debug mode that shows what task is currently running
Debug panel prompt
Add a hidden developer/debug panel opened with [SHORTCUT].

It should be lightweight and include:
- App version
- Current mode/state
- Last 50 log events
- Current task running
- Memory usage if available
- Loaded item count
- Cache size
- Recent errors
- Buttons to export logs, clear cache, reset settings, and run a self-test

Keep this hidden from normal users and do not clutter the main UI.
Bug report generator prompt
Add a one-click “Create Bug Report” option in the debug menu.

It should export a text file containing:
- App name/version
- Date/time
- Windows version if available
- App settings summary without secrets
- Last actions performed
- Recent logs
- Error stack traces if available
- File counts or scan summary
- Steps the user typed in, if there is a notes field

Make it safe to share internally.

Use These When Something Is Already Broken

Patch without rewriting everything
I already have an app. Do not rebuild it from scratch unless absolutely necessary.

Problem: [WHAT IS BROKEN]
Desired behavior: [WHAT SHOULD HAPPEN]

Please:
1. Identify the smallest safe patch
2. Keep existing working features intact
3. Avoid changing the visual design unless needed
4. Add logs around the broken area
5. Explain which files changed
6. Provide a quick test checklist
7. Mention any risk or limitation honestly
Use these as strong constraints, checklists, and acceptance criteria. No prompt can override safety rules, tool limits, or unavailable information, but these are designed to remove ambiguity and make the output much more likely to match what you intended.
