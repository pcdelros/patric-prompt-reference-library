GUI and Design System Prompts

The big upgrade: instead of asking for “Apple-like” or “modern,” ask for a design system with layout rules, spacing, components, states, and interaction behavior.
Design token system
Create the GUI using a clear design-token system instead of random styling.

Define and use tokens for:
- Background colors
- Surface/panel colors
- Text colors
- Border colors
- Accent colors
- Font sizes
- Font weights
- Spacing scale
- Corner radius
- Shadows
- Animation timing

The UI should feel polished, consistent, and easy to change later. Do not scatter hard-coded visual values everywhere. Keep the design modern, clean, and lightweight.
Component inventory prompt
Before building the GUI, define the component inventory.

Include reusable components for:
- Main window shell
- Custom title bar
- Sidebar/navigation
- Toolbar
- Search field
- Primary and secondary buttons
- Icon buttons
- Cards/panels
- Empty states
- Loading states
- Error messages
- Modal dialogs
- Toast notifications
- Progress bars
- Settings rows

Then build the app using these components consistently.
UI state behavior prompt
For every interactive element in the GUI, include visual states:
- Default
- Hover
- Pressed
- Focused
- Disabled
- Loading
- Selected
- Error

Do not use letters or weird symbols for window controls. Use clean drawn line icons for minimize, maximize/restore, and close. Keep the title bar buttons visually balanced and not oversized.
Responsive desktop layout prompt
Make the app window resizable and responsive.

Rules:
- Nothing should trail off the right side.
- Panels should scroll vertically when content is long.
- Controls should wrap or stack cleanly on smaller windows.
- Use minimum widths for dense panels.
- Avoid fixed pixel layouts that break when resized.
- Main content should remain usable at smaller sizes.
- The app should appear in the taskbar unless there is a specific reason not to.
- The minimize, maximize, restore, close, and drag behaviors should feel normal for Windows.
Accessibility without making it ugly
Make the GUI accessible while keeping it visually polished.

Include:
- Clear readable contrast
- Keyboard navigation for major actions
- Focus outlines that look intentional
- Tooltips for unclear icons
- Plain-language labels
- No tiny click targets
- No controls hidden only by color
- Friendly empty and error states
- Reduced-motion behavior if animations are heavy

Keep it clean and modern, not clinical.
Microinteraction prompt
Add lightweight microinteractions that improve clarity without making the app feel slow.

Use subtle transitions for:
- Button hover/press
- Panel opening/closing
- Loading completion
- Toast messages
- Selection changes
- Progress updates

Avoid heavy animations, constant motion, or anything that hurts performance.
GUI QA audit prompt
After creating the GUI, audit it against this checklist and fix anything that fails:
- No overlapping text
- No cutoff controls
- No weird symbols for title bar buttons
- No horizontal trailing in settings/customize panels
- Resizable window works
- Scrollable panels work
- Progress is visible during long tasks
- Error messages are understandable
- Buttons clearly describe what they do
- User-facing text avoids technical jargon
- Debug details are available but hidden from normal users
Use these as strong constraints, checklists, and acceptance criteria. No prompt can override safety rules, tool limits, or unavailable information, but these are designed to remove ambiguity and make the output much more likely to match what you intended.
